# A/B‑тест новой ленты рекомендаций: проверка боем

Проект посвящён проверке гипотезы о том, что новый алгоритм рекомендаций в развлекательном приложении с «бесконечной» лентой показывает пользователям более интересный контент. Эффективность алгоритма оценивали через A/B‑тест: сравнивали долю успешных первых сессий (где пользователь просмотрел ≥ 4 страниц) в контрольной (A) и тестовой (B) группах.

**Дата:** 07.05.2026

**Статус:** завершён.

**Стек:** Python (pandas, matplotlib, statsmodels) и статистические методы (расчёт мощности теста, z‑тест пропорций).

**Данные:**
- Исторические данные по сессиям пользователей (2025‑08‑15 — 2025‑09‑23).
- Данные за первый день A/B‑теста (2025‑10‑14).
- Данные за весь период A/B‑теста (2025‑10‑14 — 2025‑11‑02).

**Цели:**
- Проверить, повышает ли новый алгоритм рекомендаций вовлечённость пользователей.
- Оценить статистическую значимость различий в доле успешных первых сессий между группами A и B.
- Сформировать обоснованную рекомендацию по внедрению алгоритма.

**Задачи:**
- Проанализировать исторические данные: динамику регистраций, поведение пользователей в первых сессиях, базовый уровень вовлечённости.
- Рассчитать размер выборки и длительность теста.
- Проконтролировать корректность проведения теста (равномерность распределения, отсутствие пересечений пользователей).
- Сравнить ключевые метрики между группами и сделать статистический вывод.
- Сформулировать практические рекомендации.

**Ключевые этапы и решение:**
| Этапы | Альтернативное решение | Моё решение | Пояснение |
|---|---|---|---|
| **1. Загрузка и первичная проверка исторических данных** | Можно было сразу строить графики, не проверяя типы данных и пропуски. Либо использовать SQL для первичной агрегации. | Загрузила данные в pandas, проверила `.info()`, привела `session_date` к `datetime`, убедилась в структуре. | Это критично для корректной группировки по датам и избежания ошибок в дальнейшем. |
| **2. Анализ числа регистраций по дням** | Можно считать регистрацию по факту наличия сессии у зарегистрированного пользователя без привязки к дню регистрации; либо накапливать кумулятивно. | Считала уникальных пользователей и число регистраций **по дню сессии**, без накопления. Добавила столбец доли регистраций. | Так мы видим именно дневную динамику поведения, а не «накопленный итог», что важно для понимания трендов и сезонности. |
| **3. Определение прокси‑метрики вовлечённости** | В качестве метрики можно было взять среднее число страниц, время сессии, число повторных визитов. | Выбрала долю первых сессий с `page_counter >= 4` как прокси‑метрику. | Исторически видно, что 3 страницы — это «база», а 4+ — уже признак интереса; такая метрика устойчива и легко интерпретируется продуктовой командой. |
| **4. Расчёт размера выборки и длительности теста** | Можно взять «на глаз» или использовать онлайн‑калькулятор без учёта реального трафика. | Рассчитала выборку через `statsmodels` (power analysis) с MDE = 3 % от базового уровня, затем оценила длительность через средний дневной трафик. | Такой подход даёт статистически обоснованный план теста и защищает от преждевременных выводов. |
| **5. Проверка баланса групп** | Можно ограничиться сравнением общего числа строк в датасете. | Проверила: (1) разницу в числе уникальных пользователей по группам, (2) пересечения между A и B, (3) баланс по устройствам и регионам. | Проверка по уникальным пользователям (а не сессиям) и по ключевым сегментам даёт уверенность в чистоте эксперимента. |
| **6. Подготовка данных для итогового сравнения** | Можно анализировать все сессии или брать «любые» сессии пользователя. | Отфильтровала только **первые сессии** и создала флаг `good_session` (1 при `page_counter >= 4`, иначе 0). | Первые сессии лучше всего отражают «первое впечатление» от ленты и влияние алгоритма на нового/свежего пользователя; это снижает шум от лояльных пользователей. |
| **7. Статистическая проверка гипотезы** | Можно сравнить доли «на глаз» и остановиться на этом; либо использовать t‑тест для средних. | Применила **z‑тест пропорций** для сравнения долей успешных первых сессий, сформулировала одностороннюю альтернативу. | Доля — бинарная метрика, поэтому z‑тест пропорций корректнее, чем t‑тест. Односторонняя альтернатива соответствует бизнес‑гипотезе «новый алгоритм лучше». |
| **8. Интерпретация результатов** | Можно сделать вывод «группа B чуть лучше», игнорируя p‑value. | Оценила p‑value относительно уровня значимости α = 0.05 и сделала вывод о статистической значимости. | Это защищает от принятия случайных колебаний за эффект алгоритма. |

**Выводы:**
- Целевая метрика - доля успешных первых сессий.
- Группы сбалансированы по размеру, нет пересечений пользователей, распределение по устройствам и регионам сопоставимо.
- Нулевая гипотеза ($H_0$): доли успешных первых сессий в группах A и B значимо не различаются ($p_A = p_B$).
- Альтернативная гипотеза ($H_1$): доля успешных первых сессий в группе B выше, чем в группе A ($p_B > p_A$).
- Нет оснований отвергать $H_0$. Статистически значимого эффекта от нового алгоритма на выбранную метрику не обнаружено.

**Рекомендации:**
- Не внедрять новый алгоритм в текущем виде на основе этой метрики.
- Расширить набор метрик.
- Перепроверить дизайн теста.
